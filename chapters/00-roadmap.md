# 学習ロードマップと到達基準（2年目を目指す）

このカリキュラムは、**全くの初学者が、案件に参画して価値提供できる「Android開発2年目レベル」**に到達することを目的にしています。
Compose中心・DIはHilt・APIはRESTを前提に、実務で要求される「作る力」だけでなく、**壊さずに直す力**（調査・テスト・運用・チーム開発）までカバーします。

---

## このカリキュラムのゴール（2年目の定義）

以下が「できる」と、実務のチケットを任せられる状態に近づきます。

- [ ] 既存アプリの構成（UI/VM/Repository/DB/Network）を読み、**変更すべき箇所を特定**できる
- [ ] Composeで画面を追加し、状態/遷移/エラー表示を含めて実装できる
- [ ] REST API連携（Retrofit + OkHttp）を組み込み、失敗時の挙動（再試行/エラー表示）まで作れる
- [ ] Room/DataStoreでデータを永続化し、オフラインでも破綻しない設計にできる
- [ ] Hiltで依存関係を整理し、テスト用に差し替えられる
- [ ] テスト（ユニット/Compose UI）を追加し、**回帰を防ぐ**形で修正できる
- [ ] 不具合を再現し、Logcat/Debugger/Profilerで原因を絞り、再発防止策まで提案できる
- [ ] PRを作り、説明・スクショ・テスト結果を揃えてレビューを通せる

---

## 進め方（最短で伸びるやり方）

このカリキュラムは「読む」だけで完走しない設計にしてください。おすすめは以下です。

1. 各章で作るものを、手元のサンプルアプリ（シンプルメモ）に必ず実装する
2. 章末のチェックリストを、**“できる証拠”**（PR/スクショ/テスト）として残す
3. 1つ進むごとに「なぜその設計にしたか」を1〜3行でメモする（後からレビューで効く）

---

## 卒業要件（チェックリスト）

### A. 開発環境とビルド

- [ ] 新規PCでもAndroid Studio/SDK/エミュレータを整備できる
- [ ] Gradle同期/依存関係更新のトラブルを自力で切り分けできる
- [ ] Debug/Releaseの違いを理解し、署名・バージョン更新ができる

### B. UI（Compose）

- [ ] 状態（State）とUIを分離し、UIは描画に専念させられる
- [ ] 画面遷移（Navigation）を安全に扱える（戻る・DeepLink含む）
- [ ] リスト/フォーム/ダイアログ/スナックバー等の実装ができる
- [ ] 画面サイズ（スマホ/タブレット）で崩れない設計ができる

### C. データ層（Network / DB / 非同期）

- [ ] Retrofit/OkHttpでREST APIを呼べる（失敗/タイムアウトを含む）
- [ ] Coroutines/Flowを正しく使い、UIスレッドをブロックしない
- [ ] RoomでCRUDができ、データソースをRepositoryに隠蔽できる

### D. アーキテクチャと品質

- [ ] MVVM + UI Stateパターンで画面を設計できる
- [ ] Hiltで依存を解決し、テストでFakeに差し替えられる
- [ ] ViewModel/Repositoryにユニットテストを追加できる
- [ ] 重要な画面はCompose UIテストで最低限守れる

### E. 実務の進め方（2年目の生命線）

- [ ] 変更前に「再現手順」「期待結果」「影響範囲」を書ける
- [ ] PRに、要点・スクショ・テスト結果・リスクを書ける
- [ ] レビューで指摘されやすい点（命名、責務、境界、例外）を先回りできる

## 模擬案件（最小セット）

以下の3本を通すと、参画後の“初速”が出ます。

### チケット1：機能追加（UI + API + 永続化）

- 例：メモ一覧に「検索」「フィルタ」「並び替え」を追加し、設定をDataStoreに保存する
- 受け入れ条件：空状態/エラー時表示、回転後も状態保持、最低1つはテスト追加

### チケット2：不具合修正（調査 + 回帰防止）

- 例：特定操作でクラッシュする/二重送信される/画面が真っ白になる
- 受け入れ条件：再現手順、原因、修正方針、回帰テスト or 再発防止策

### チケット3：リファクタ（壊さず整える）

- 例：Repositoryの責務分離、例外設計の統一、マルチモジュール化の第一歩
- 受け入れ条件：差分最小、動作確認手順、影響範囲、段階移行の説明

---

## AIの使い方（実務寄り）

AIは「答えを出す」より、**調査の質を上げる**用途で使うと強いです。

- 「このスタックトレースから原因候補を3つ挙げて、切り分け手順も出して」
- 「このViewModelの責務が肥大化している。分割案とテスト戦略を提案して」
- 「このPR説明文を、レビュアーが判断しやすい形に整えて」
